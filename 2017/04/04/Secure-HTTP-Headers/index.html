HTTP安全相关的Headers

前两天颖聪share了一篇关于HTTPS优化的topic，其中涉及到了HTTP Strict Transport Security(HSTS)相关的内容，个人比较感兴趣，所以稍微花了点时间做了些扩展阅读。这篇文章会简单介绍一些HTTP安全相关的请求/响应头。

HTTP Strict Transport Security (HSTS)

HTTP Strict Transport Security（HSTS）是由IETF发布的互联网安全策略机制。通过添加Strict-Transport-Security header来强制浏览器使用HTTPS与网站进行通信，从而减少中间人攻击(MitM)的风险。

refs:
HTTP Strict Transport Security - 安全 | MDN

X-XSS-Protection

正如其名，这个头是用来防止Cross-site Scripting (XSS)攻击的，它会强制浏览器对HTTP的请求/响应做检查，但检查的机制比较简单粗暴：寻找符合XSS攻击的code pattern，因此有一定的误判率。另外，firefox并不支持这个Header。

1 or 0：表示enable 或 disable；mode=block表示当检测到XSS攻击，则block页面渲染。

refs:
X-XSS-Protection - HTTP | MDN

X-Frame-Options

X-Frame-Options规定了当前页面能否被iframe到别的网页中。developer可以设置这个Header来限制或者阻止其他网站通过iframe/frame/object等标记来内嵌你的网页，从而避免了clickjacking的攻击。

Cache-Control

下面是几个选项的官方定义：
no-cache：no-cache 不意味着 “别缓存”，它强制所有缓存了该响应的用户，在使用已存储的缓存数据前，发送带验证器的请求到原始服务器；
no-store：不可缓存客户端请求或服务端响应；
must-revalidate: must-revalidate也不意味着 “必须做检查”，它意味着，如果本地的缓存没有过期（没超过max-age），则可以使用，否则必须重新验证。

Progma是HTTP1.0里的Header。
no-cache: 和Cache-Control里的no-cache一样。

Expires: 这里设置的是响应的过期时间，因为设置了-1，所以当浏览器接收完响应，就会立刻把这个响应作为过期来处理，从而避免了缓存。
